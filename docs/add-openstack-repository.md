# Adding an OpenStack Repository

This guide describes how to add another OpenStack repository, such as Keystone, to the Repointel ingestion and review-risk pipeline.

The important rule: do not duplicate normalizer rules per repository. Reuse the shared normalizer, and keep only source-specific settings on each source.

## Current Model

Each repository needs one `repositories` row and usually three `sources` rows:

- Gerrit reviews: provider `gerrit`, type `code_reviews`
- Git commits: provider `git`, type `commits`
- Launchpad bugs: provider `launchpad`, type `bugs`

All three sources should point at the same shared normalizer with `normalizer_id`.

For future GitHub support, this shape should stay the same: add provider-specific sources, but keep reusable metadata rules in shared normalizers.

## Required Inputs

For an OpenStack repo, collect:

```json
{
  "project": "openstack/keystone",
  "repo_slug": "keystone",
  "repo_name": "openstack/keystone",
  "git_url": "https://opendev.org/openstack/keystone",
  "gerrit_base_url": "https://review.opendev.org",
  "launchpad_project": "keystone",
  "local_path": "/media/tb/linux/home/tim/dev/openstack/keystone",
  "default_branch": "master"
}
```

Verify `local_path` contains a `.git` directory. Git ingestion reads from local disk, so this path is required for fast commit ingestion and line-survival analysis.

## Shared Normalizer

Use the existing shared metadata normalizer. Do not create a Keystone copy.

Find it with:

```sql
select id, doc->>'name', doc->>'version'
from repointel_records
where collection = 'normalizers';
```

Every new OpenStack source should set:

```json
{
  "normalizer_id": "<shared-normalizer-id>"
}
```

The shared normalizer should have a generic OpenStack name. If it still has a repository-specific display name, rename the existing row through `PATCH /normalizers/{normalizer_id}` instead of creating a copy.

## Repository Row

Create one repository row:

```json
{
  "repository_group_id": "<openstack-group-id>",
  "slug": "keystone",
  "name": "openstack/keystone",
  "vcs": "git",
  "canonical_url": "https://opendev.org/openstack/keystone",
  "default_branch": "master",
  "status": "active"
}
```

Store the returned `repository_id`; each source uses it.

## Gerrit Source

Create one Gerrit source:

```json
{
  "repository_id": "<repository-id>",
  "name": "Gerrit reviews",
  "type": "code_reviews",
  "provider": "gerrit",
  "base_url": "https://review.opendev.org",
  "external_key": "openstack/keystone",
  "enabled": true,
  "normalizer_id": "<shared-normalizer-id>",
  "ingestion_policy": {
    "limit": 1000,
    "review_limit": 1000,
    "reviews_per_minute": 6,
    "comments_per_change": 10000,
    "include_automated_messages": false
  },
  "ingestion_filters": {}
}
```

Do not copy watermarks such as `gerrit_updated_watermark` from another repository. They are per-source runtime state.

## Git Source

Create one git source:

```json
{
  "repository_id": "<repository-id>",
  "name": "Commit messages",
  "type": "commits",
  "provider": "git",
  "base_url": "https://opendev.org/openstack/keystone",
  "external_key": "openstack/keystone",
  "enabled": true,
  "normalizer_id": "<shared-normalizer-id>",
  "ingestion_policy": {
    "limit": 20000,
    "local_path": "/media/tb/linux/home/tim/dev/openstack/keystone",
    "line_survival_enabled": true,
    "line_survival_branch": "origin/master",
    "line_survival_window_days": 365,
    "line_survival_max_changed_lines": 100,
    "approval_line_survival_branch": "origin/master",
    "approval_line_survival_max_review_changed_lines": 200
  },
  "ingestion_filters": {}
}
```

Only these line-survival fields are configuration:

- `line_survival_enabled`
- `line_survival_branch`
- `line_survival_window_days`
- `line_survival_max_changed_lines`
- `approval_line_survival_branch`
- `approval_line_survival_max_review_changed_lines`

Do not copy these fields from another repo:

- `line_survival_head_sha`
- `line_survival_config_key`
- `line_survival_analyzed_at`
- `line_survival_since_epoch`
- `line_survival_commits_*`
- `approval_line_survival_head_sha`
- `approval_line_survival_config_key`
- `approval_line_survival_fingerprint`
- `approval_line_survival_analyzed_at`
- `approval_line_survival_since_epoch`
- `approval_line_survival_commits_*`
- `approval_line_survival_reviewers_analyzed`
- `approval_line_survival_approvals_*`

Those are generated by ingestion for that specific repository.

## Launchpad Source

Create one Launchpad source:

```json
{
  "repository_id": "<repository-id>",
  "name": "Launchpad bugs",
  "type": "bugs",
  "provider": "launchpad",
  "base_url": "https://api.launchpad.net/1.0/keystone",
  "external_key": "keystone",
  "enabled": true,
  "normalizer_id": "<shared-normalizer-id>",
  "ingestion_policy": {
    "limit": 1000
  },
  "ingestion_filters": {}
}
```

Do not copy `launchpad_updated_watermark` from another repository.

## Bring Current

After the repo and sources exist, use repository sync:

```http
POST /repositories/<repository_id>/enqueue-ingestion
Authorization: Bearer writer
Content-Type: application/json
```

```json
{
  "mode": "repository-sync",
  "requested_by": "operator",
  "params": {
    "skip_unchanged_raw_records": true,
    "run_normalizers_for_new_records_only": true
  }
}
```

The server should expand this to all enabled sources for the repository.

Normal sync should process only new or changed raw records. If new normalizer rules are added later, run a separate explicit full metadata reprocess instead of doing that on every sync.

## Review Commit SHA Metadata

Merged Gerrit reviews should expose their final commit SHA as review metadata, not only inside raw Gerrit JSON. The shared normalizer should include:

- `code_review.gerrit/current_revision_sha` from `raw.payload.current_revision`
- `code_review.gerrit/current_revision_number` from `raw.payload.current_revision_number`

Backfill existing merged reviews after adding or repairing those rules:

```bash
node projects/repointel-metadata-collection/tools/backfill_gerrit_review_commit_metadata.mjs
```

The backfill is idempotent and defaults to merged Gerrit changes only. Use `--repository-id <repository-id>` to scope it to one repository, and `--dry-run` to preview counts.

## Expected Order

For best results:

1. Ingest Gerrit reviews.
2. Ingest git commits.
3. Ingest Launchpad bugs.
4. Run or allow normalizers for new raw records.
5. Run ONNX scoring for new text if the scorer is configured.
6. Run line-survival and approval-survival analysis from the git source.
7. Open Review Risk filtered to the repository or project.

Approval survival needs both Gerrit approval metadata and git commit stats. If either side is missing, the review-risk page still works, but reviewer survival signals will be sparse.

## Backfill Approval Survival

If approval-survival metadata is missing or stale, regenerate it from Gerrit raw records and a line-survival JSON file:

```bash
python3 projects/repointel-metadata-collection/tools/git_line_survival.py \
  --repo /media/tb/linux/home/tim/dev/openstack/keystone \
  --branch origin/master \
  --max-changed-lines 200 \
  --json-out projects/repointel-metadata-collection/artifacts/keystone-line-survival-approval200.json \
  --markdown-out projects/repointel-metadata-collection/artifacts/keystone-line-survival-approval200.md

node projects/repointel-metadata-collection/tools/backfill_gerrit_approval_survival.mjs \
  --repository-id <repository-id> \
  --line-survival-json projects/repointel-metadata-collection/artifacts/keystone-line-survival-approval200.json \
  --max-review-changed-lines 200
```

The backfill reads positive Gerrit `Code-Review+1/+2` messages from raw review records, joins them to the merged commit SHA, and writes `review.approval_line_survival` metadata keyed by Gerrit account. It replaces old derived approval-survival metadata for that repository so the UX does not double-count partial rows.

## Validation Queries

Check repo and sources:

```sql
select id, doc->>'slug', doc->>'canonical_url'
from repointel_records
where collection = 'repositories'
order by doc->>'slug';

select id, doc->>'repository_id', doc->>'provider', doc->>'type',
       doc->>'external_key', doc->>'normalizer_id'
from repointel_records
where collection = 'sources'
order by doc->>'repository_id', doc->>'provider';
```

Check ingestion output:

```sql
select doc->>'source_id' as source_id,
       count(*) filter (where collection = 'raw-records') as raw_records,
       count(*) filter (where collection = 'arts') as arts
from repointel_records
where collection in ('raw-records', 'arts')
group by doc->>'source_id'
order by source_id;
```

Check metadata coverage:

```sql
select doc->>'namespace' as namespace, doc->>'key' as key, count(*)
from repointel_records
where collection = 'metadata'
group by 1, 2
order by count(*) desc
limit 50;
```

## What Should Be DRY

Shared:

- metadata normalizer rules
- review concern vocabulary
- security keyword categories
- ONNX scorer configuration
- default OpenStack source policy template

Per repository:

- repository row
- source rows
- Gerrit `external_key`
- Launchpad project URL/key
- git `local_path`
- source watermarks and cursors
- line-survival generated state

If a setup tool is added, it should take the required inputs once and create the repository plus provider sources, all pointing to the shared normalizer.
